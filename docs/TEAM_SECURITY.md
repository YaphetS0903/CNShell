# CNshell 团队工作区安全模型

当前版本提供本机团队工作区、成员、RBAC、密钥 epoch 和元数据审计基础。它用于固定客户端
权限语义和后续端到端分享协议，不等同已经上线的账号、邀请或多人终端服务。

## 设备身份与安全分享

每个本机工作区设备生成独立的 X25519 加密密钥和 Ed25519 签名密钥。私钥只保存在
`com.cnshell.team-device` Keychain 服务中；SQLite 和 `.cnshelldevice` 公钥文件只包含公钥。
导入远端设备前必须通过独立渠道核对双方公钥指纹。

`.cnshellshare` v1 使用以下封装：

1. 每个分享生成随机 32 字节内容密钥和临时 X25519 密钥。
2. 连接载荷使用 AES-256-GCM 加密，AAD 绑定工作区、分享 ID 和密钥 epoch。
3. 对每台接收设备分别执行 X25519，再用 HKDF-SHA-256 派生独立封装密钥，以 AES-256-GCM
   封装同一个内容密钥。接收者列表最多 64 台，错误设备不能解封。
4. 发送设备用 Ed25519 签署 RFC 8785 规范化后的完整 envelope；签名覆盖 epoch、发送者、
   全部接收者、nonce、封装密钥和连接密文。
5. 接收端先验证活动成员与设备、签名和 epoch，再解密并展示不含秘密的预览。预览 5 分钟
   过期且只能使用一次；确认后凭据直接写入新的 Keychain 项，不返回前端。

首版只分享无代理、无私钥/证书文件的 SSH/RDP 连接。载荷不包含备注、启动命令、环境变量、
本地路径或代理；密码只有 Owner/Admin 具备 `shareManage` 且再次确认时才加入密文。分享明文
最多 1 MB、文件最多 4 MB，输入拒绝符号链接。设备撤销和成员移除会推进 epoch，并阻止该
设备继续创建或打开分享。

## 角色与权限

| 权限 | Owner | Admin | Operator | Viewer |
| --- | --- | --- | --- | --- |
| 查看工作区、成员和连接元数据 | 是 | 是 | 是 | 是 |
| 查看共享终端 | 是 | 是 | 是 | 是 |
| 使用连接、控制共享终端 | 是 | 是 | 是 | 否 |
| 创建分享 | 是 | 是 | 是 | 否 |
| 管理连接 | 是 | 是 | 否 | 否 |
| 管理普通成员 | 是 | 是 | 否 | 否 |
| 管理 Owner | 是 | 否 | 否 | 否 |
| 查看、导出审计 | 是 | 是 | 否 | 否 |
| 删除工作区 | 是 | 否 | 否 | 否 |

Owner、Admin、Operator 和 Viewer 的权限表只在 Rust 后端定义，前端隐藏按钮不是授权边界。
所有成员变更都由后端重新读取本机成员角色并在同一 SQLite 事务内检查和写入。

## 不变量

- 每个工作区至少保留一名活动 Owner。
- Admin 不能新增、降级或移除 Owner。
- 本机成员不能通过本地界面移除自己，必须先完成 Owner 和设备交接。
- 一个成员 UUID 不能跨工作区复用，避免冲突更新触碰其他工作区。
- 成员移除、恢复或角色变化会递增工作区 `keyEpoch`。后续加密分享只能使用当前 epoch
  创建新内容密钥，已移除成员不能获得后续 epoch 的密钥封装。
- 设备撤销同样递增 epoch；已撤销的发送或接收设备均不能通过客户端校验。旧 epoch 分享只
  对仍活动、当时已列为接收者且仍持有私钥的设备可读。
- 删除成员采用 `removed` 状态，不删除历史审计。

## 审计

团队审计只记录：工作区 ID、执行成员 ID、动作、目标类型、目标 ID 和时间。不会记录命令、
终端正文、主机地址、用户名、路径或凭据。每个工作区最多保留 4,096 条，导出使用同目录
临时文件、`fsync` 和原子改名；文件成功落盘后才追加 `audit-exported` 事件。

## 多人终端协议基础

CNshell 已实现不依赖具体 relay 的主持端状态机，但尚未连接正式网络服务。房间只可绑定当前
真实 SSH 会话，并要求主持端是当前工作区活动 Owner/Admin 和活动本机设备。参与设备最多
64 台，加入时必须仍属于活动成员并具备 `terminalView`。

- 输出拆为 1–64 KB 帧，房间内序号从 1 严格递增。
- 所有参与者默认只读。只有具备 `terminalControl` 的 Owner/Admin/Operator 可获得租约。
- 控制权同一时刻只有一个设备持有，租约 10–300 秒；新租约替换旧租约并递增 generation。
- 每个输入帧重新校验主持成员/设备/epoch、控制成员角色、设备状态、租约 ID/归属/有效期和
  设备单调输入序号。重复、跳号、过期、角色降级、成员移除或设备撤销会立即拒绝。
- 单输入帧最多 64 KB 且必须为 UTF-8；后端校验通过后才写入原 SSH channel。审计只记录
  房间开始/关闭、参与设备和租约授予/撤销，不记录终端正文或输入内容。

当前 `TeamTerminalFrame` 是主持进程内部的明文 IPC 帧，不能直接发送到不受信任网络。正式
relay 适配器必须先用会话内容密钥加密、绑定房间/方向/序号作为 AAD，并由发送设备签名；
relay 只能看到有界密文和路由 ID。完成该适配器、邀请密钥封装、断线恢复与双设备测试前，
UI 不提供“在线多人终端”入口。

## 正式服务端要求

本机数据库属于当前 macOS 用户的信任域，不能替代服务端授权。正式团队服务必须：

1. 在每个 API 和 WebSocket 消息上重复执行相同或更严格的 RBAC，不能信任客户端提交的角色。
2. 使用短期、可撤销的设备会话，并把组织、工作区、成员和设备 ID 绑定到签名令牌。
3. 只存储端到端密文、每设备密钥封装和最小路由元数据，不能接收连接凭据明文。
4. 成员移除或设备撤销后拒绝当前及后续 epoch 请求；轮换必须防回滚和重放。
5. 多人终端默认只读，控制权采用单持有者租约、明确移交、超时回收和 Owner/Admin 强制回收。
6. 服务端审计同样不能记录完整终端输出或秘密，并提供组织保留、导出和删除策略。

在这些服务端条件和真实多设备验收完成前，CNshell UI 明确标记团队能力为本机基础，不声明
在线邀请、同步、多人终端或成员移除后的服务端阻断已经完成。
