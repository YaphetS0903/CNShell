# CNshell 团队工作区安全模型

当前版本提供本机团队工作区、成员、RBAC、密钥 epoch 和元数据审计，并包含独立的在线
relay 服务端实现。服务端代码和本机 loopback 验收不等同已经部署正式域名、TLS、邮件或
真实跨设备多人终端服务。

## 设备身份与安全分享

每个本机工作区设备生成独立的 X25519 加密密钥和 Ed25519 签名密钥。私钥只保存在
`com.cnshell.team-device` Keychain 服务中；SQLite 和 `.cnshelldevice` 公钥文件只包含公钥。
导入远端设备前必须通过独立渠道核对双方公钥指纹。

`.cnshellshare` v1 使用以下封装：

1. 每个分享生成随机 32 字节内容密钥和临时 X25519 密钥。
2. 连接载荷使用 AES-256-GCM 加密，AAD 绑定工作区、分享 ID 和密钥 epoch。
3. 对每台接收设备分别执行 X25519，再用 HKDF-SHA-256 派生独立封装密钥，以 AES-256-GCM
   封装同一个内容密钥。接收者列表最多 64 台，错误设备不能解封。
4. 发送设备用 Ed25519 签署 RFC 8785 规范化后的完整 envelope；签名覆盖 epoch、发送者、
   全部接收者、nonce、封装密钥和连接密文。
5. 接收端先验证活动成员与设备、签名和 epoch，再解密并展示不含秘密的预览。预览 5 分钟
   过期且只能使用一次；确认后凭据直接写入新的 Keychain 项，不返回前端。

首版只分享无代理、无私钥/证书文件的 SSH/RDP 连接。载荷不包含备注、启动命令、环境变量、
本地路径或代理；密码只有 Owner/Admin 具备 `shareManage` 且再次确认时才加入密文。分享明文
最多 1 MB、文件最多 4 MB，输入拒绝符号链接。设备撤销和成员移除会推进 epoch，并阻止该
设备继续创建或打开分享。

## 角色与权限

| 权限 | Owner | Admin | Operator | Viewer |
| --- | --- | --- | --- | --- |
| 查看工作区、成员和连接元数据 | 是 | 是 | 是 | 是 |
| 查看共享终端 | 是 | 是 | 是 | 是 |
| 使用连接、控制共享终端 | 是 | 是 | 是 | 否 |
| 创建分享 | 是 | 是 | 是 | 否 |
| 管理连接 | 是 | 是 | 否 | 否 |
| 管理普通成员 | 是 | 是 | 否 | 否 |
| 管理 Owner | 是 | 否 | 否 | 否 |
| 查看、导出审计 | 是 | 是 | 否 | 否 |
| 导出组织目录 | 是 | 是 | 否 | 否 |
| 删除工作区 | 是 | 否 | 否 | 否 |

Owner、Admin、Operator 和 Viewer 的权限表只在 Rust 后端定义，前端隐藏按钮不是授权边界。
所有成员变更都由后端重新读取本机成员角色并在同一 SQLite 事务内检查和写入。

## 不变量

- 每个工作区至少保留一名活动 Owner。
- Admin 不能新增、降级或移除 Owner。
- 本机成员不能通过本地界面移除自己，必须先完成 Owner 和设备交接。
- 一个成员 UUID 不能跨工作区复用，避免冲突更新触碰其他工作区。
- 成员移除、恢复或角色变化会递增工作区 `keyEpoch`。后续加密分享只能使用当前 epoch
  创建新内容密钥，已移除成员不能获得后续 epoch 的密钥封装。
- 设备撤销同样递增 epoch；已撤销的发送或接收设备均不能通过客户端校验。旧 epoch 分享只
  对仍活动、当时已列为接收者且仍持有私钥的设备可读。
- 删除成员采用 `removed` 状态，不删除历史审计。

## 审计

团队审计只记录：工作区 ID、执行成员 ID、动作、目标类型、目标 ID 和时间。不会记录命令、
终端正文、主机地址、用户名、路径或凭据。每个工作区最多保留 4,096 条，导出使用同目录
临时文件、`fsync` 和原子改名；文件成功落盘后才追加 `audit-exported` 事件。

Owner/Admin 可另行导出组织目录。该原子 JSON 只包含工作区、成员、设备公钥/指纹和上述
元数据审计，不包含连接、主机、凭据、token 或终端内容；后端拒绝符号链接目标，文件成功
落盘后才追加 `workspace-exported` 事件。

## 多人终端协议与在线接入

CNshell 已实现不依赖具体 relay 的端到端状态机，并把它接入受管在线 WebSocket。房间只可
绑定当前真实 SSH 会话，并要求主持端是当前工作区活动 Owner/Admin 和活动本机设备。参与
设备最多 64 台，加入时必须仍属于活动成员并具备 `terminalView`。

- 输出拆为 1–64 KB 帧，房间内序号从 1 严格递增。
- 所有参与者默认只读。只有具备 `terminalControl` 的 Owner/Admin/Operator 可获得租约。
- 控制权同一时刻只有一个设备持有，租约 10–300 秒；新租约替换旧租约并递增 generation。
- 每个输入帧重新校验主持成员/设备/epoch、控制成员角色、设备状态、租约 ID/归属/有效期和
  设备单调输入序号。重复、跳号、过期、角色降级、成员移除或设备撤销会立即拒绝。
- 单输入帧最多 64 KB 且必须为 UTF-8；后端校验通过后才写入原 SSH channel。审计只记录
  房间开始/关闭、参与设备和租约授予/撤销，不记录终端正文或输入内容。

本机兼容命令中的 `TeamTerminalFrame` 不能直接发送到网络；在线适配器只发送独立的 v1
加密 envelope：

1. 每个房间生成随机 32 字节内容密钥，只保存在主持端和已接受邀请客户端的进程内存。
2. 邀请使用临时 X25519 与 HKDF-SHA-256 派生接收设备专属封装密钥，再以 AES-256-GCM
   封装房间密钥；AAD 绑定工作区、房间、epoch 和接收设备。主持设备对 RFC 8785 规范化
   后的完整邀请执行 Ed25519 签名，邀请 5 分钟过期且不能重复接受来回滚序号。
3. 输出和控制输入均以 AES-256-GCM 加密，并由实际发送设备 Ed25519 签名。AAD 绑定工作区、
   房间、epoch、发送成员/设备、方向、类型、序号、租约 ID 与 generation；relay 只能看到
   有界密文和路由元数据，无法伪造正文或把帧换到其他房间、方向或租约。
4. 接收端在验签和解密前重新读取活动成员、设备、角色与当前 epoch。输出和每设备输入序号
   必须严格连续；篡改、重复、跳号、旧 epoch、角色降级、设备撤销和旧控制租约都会拒绝。
5. 主持端只在内存保留 5 分钟、最多 512 帧/4 MB 的输出密文，用于客户端按最后已接收序号
   补帧。窗口不足时要求重新加入，不把终端正文或房间密钥写入 SQLite、审计或 relay。

上述加密传输层已使用两份独立客户端数据库和各自 Keychain 设备密钥完成本机回环测试；
历史分享测试还验证 epoch 轮换后仍活动的原接收设备可以解密，而撤销和篡改继续拒绝。
relay 服务端也已完成双账号双设备 HTTP/WebSocket 测试。客户端账号、设备 challenge、工作区
邀请、权威目录同步、房间 WebSocket 和观看/控制 UI 均已接通。握手同步服务端游标、真实已
加入成员和当前租约；参与者离开会撤销访问和其租约。生产 WSS 部署和两台真实设备的网络
验收仍未完成，因此入口只用于 loopback 或用户自行部署的测试 relay。

## 正式服务端要求

本机数据库属于当前 macOS 用户的信任域，不能替代服务端授权。正式团队服务必须：

1. 在每个 API 和 WebSocket 消息上重复执行相同或更严格的 RBAC，不能信任客户端提交的角色。
2. 使用短期、可撤销的设备会话，并把组织、工作区、成员和设备 ID 绑定到签名令牌。
3. 只存储端到端密文、每设备密钥封装和最小路由元数据，不能接收连接凭据明文。
4. 成员移除或设备撤销后拒绝当前及后续 epoch 请求；轮换必须防回滚和重放。
5. 多人终端默认只读，控制权采用单持有者租约、明确移交、超时回收和 Owner/Admin 强制回收。
6. 服务端审计同样不能记录完整终端输出或秘密，并提供组织保留、导出和删除策略。

`services/team-relay` 已在代码层落实上述六项：每个请求和帧重新授权、短期 opaque token、
设备 Ed25519 challenge、仅存密文、epoch/撤销传播、服务端控制租约、4,096 条元数据审计、
审计读取和 Owner 永久删除；同时提供数据库感知的 readiness、活动 WebSocket 优雅停机、
不含工作区/设备/房间标签的 Prometheus 指标、默认强制 `age` 的一致性备份和校验恢复。
详情与部署边界见 `docs/TEAM_RELAY.md` 和
`docs/TEAM_RELAY_OPERATIONS.md`。

在线账号、邀请、工作区同步和多人终端现在可配置使用，但在正式 TLS/WSS 服务、邮件验证、
限速、监控平台、生产 identity 加密异地恢复和多设备验收完成前，CNshell 不声明其为可用的
生产团队服务。
